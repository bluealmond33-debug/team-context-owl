# NotebookLM Handoff — GitHub 맥락을 질문 가능한 팀 브리핑으로 만들기

## Context Hippo가 하는 일과 하지 않는 일

Context Hippo는 GitHub/Obsidian의 자료를 **NotebookLM용 정본 한 개**로 정리하는 데 도움을 줍니다. NotebookLM에 GitHub 저장소를 자동 연결하거나, 변경을 자동 업로드하지는 않습니다.

이 구분이 중요합니다. 저장소 전체·raw 회의록·오래된 초안을 전부 올리면 NotebookLM 결과가 흐려지고, 민감한 내용을 잘못 올릴 위험도 있습니다.

## 3분 사용법

1. `99 Templates/notebooklm-handoff.md`를 복사합니다.
2. `07 Outputs/YYYY-MM-DD-notebooklm-handoff.md`로 저장합니다.
3. `_system/CURRENT_STATE.md`, `_system/DECISION_LOG.md`, 최근 변경과 직접 관련된 근거만 읽어 채웁니다.
4. 기준 커밋·포함 범위·제외한 내용을 적습니다.
5. 업로드 전 체크리스트를 통과한 Markdown 또는 PDF 한 개만 NotebookLM에 올립니다.
6. 문서 안의 **“NotebookLM에서 바로 쓸 질문”**에서 목적에 맞는 프롬프트를 복사합니다.

## 어떤 프롬프트가 있는지 어떻게 알 수 있나

프롬프트를 별도 문서에 숨기지 않습니다. **매 Handoff 패키지 안에 목적별 질문 4개를 기본 포함**합니다.

| 원하는 결과 | 패키지 안에서 고를 질문 |
|---|---|
| 이번 주 팀 공지 | `팀 브리핑` |
| 발표용 슬라이드 초안 | `발표 슬라이드` |
| 새 팀원 설명 | `온보딩 / Q&A` |
| 아직 정하지 못한 일 정리 | `의사결정 검토` |

사용자는 NotebookLM을 열기 전에 이 가이드를 외울 필요가 없습니다. 업로드하는 파일의 6절만 보면 됩니다. 이것이 가장 발견하기 쉬운 UX입니다.

## 안전한 소스 선택

### 넣기 좋은 것
- 현재 상태, 확정된 ADR, 최신 팀 합의
- 배포·데모에서 실제로 검증한 결과
- 최근 기간의 변경 요약과 원문 링크

### 기본적으로 제외할 것
- `*_raw/`의 원문 전체, 중복 회의록, 폐기된 초안
- 토큰·키·계정 정보, 개인정보, 고객 데이터
- 오래된 목표·아직 검증하지 않은 수치·실제 연동처럼 보일 수 있는 설명

## 권장 운영 방식

- **슬라이드용:** 최근 확정 사항과 발표에 필요한 근거만 담은 짧은 패키지
- **팀 Q&A용:** 배경·결정·미해결 항목까지 포함한 패키지
- **갱신 시점:** 중요한 결정, 데모 상태, 릴리스가 바뀔 때만 새 패키지 생성
- **추적:** 패키지 첫머리에 기준 commit을 남기고, 다음 패키지는 그 commit 이후의 변경만 비교

NotebookLM 생성 한도나 공유 권한은 NotebookLM 계정 설정에 따릅니다. Context Hippo 패키지는 내용을 정리하고 검토하기 쉽게 만들지만, NotebookLM의 생성·업로드·공유를 대신 실행하지 않습니다.
